# graphql-fs

Project a GraphQL SDL onto a **navigable file tree** so an LLM can explore a
large schema with ordinary shell tools (`ls`, `cat`, `grep`, `find`) instead of
holding the whole SDL in context.

The premise: models are extremely good at filesystem navigation, and a file
tree gives *progressive disclosure* — the model walks to exactly the slice of
the schema it needs. Graph edges (a field's return type, a union's members, an
interface's implementations) become **symlinks**, so the model can literally
follow the graph by walking directories.

This is a self-contained experiment that accompanies the discussion of exposing
a (super)graph to an LLM as a filesystem. It works on any plain GraphQL SDL —
no federation/subgraph metadata is needed, since prompt-to-query generation only
depends on the public schema shape.

## Build

```sh
cd graphql-fs
go build -o graphql-fs .
```

## Usage

### `build` — SDL → file tree

```sh
# From a URL (monday.com publishes its SDL unauthenticated):
graphql-fs build -sdl "https://api.monday.com/v2/get_schema?format=sdl"
# -> prints the temp dir it created

# From a file, into a chosen directory:
graphql-fs build -sdl ./schema.graphql -out ./tree
```

Layout of the generated tree:

```
schema.graphql              the complete SDL
index.txt                   type counts + every Query/Mutation/Subscription field
README.txt                  how to navigate (written into every tree)
query/<field>.graphql       one "card" per root query field (entry points)
mutation/<field>.graphql    one card per root mutation field
subscription/<field>.graphql
directives/<name>.txt
types/<TypeName>/
    type.graphql            the type's own SDL block
    meta.txt                kind, field count, description
    fields/<field>.graphql  (objects/interfaces) signature, args (+docs/defaults)
    input_fields/<f>.graphql (input objects)
    values/<value>.txt      (enums)
    references/<field>  ->   SYMLINK to the field's return-type directory
    implements/<iface>  ->   SYMLINK (object  -> interfaces it implements)
    implemented_by/<obj> ->  SYMLINK (interface -> implementing objects)
    possible_types/<obj> ->  SYMLINK (union -> member types)
```

Navigating it:

```sh
cat index.txt                                  # overview + entry points
cat query/items.graphql                        # an entry point's args + return
ls  types/Item/fields                          # what fields an Item has
cat types/Item/fields/column_values.graphql    # one field's details
ls -l types/Item/references                     # follow a field to its type
cat types/Item/references/column_values/type.graphql
grep -ril "column" types/*/fields              # search fields by keyword
```

### `compare` — grade a query against a ground-truth query

Objective, schema-aware comparison (not a string diff). Both queries are parsed
and validated against the schema; then field paths and canonicalised arguments
are compared.

```sh
graphql-fs compare -sdl ./schema.graphql \
  -expected ./expected.graphql \
  -generated ./actual.graphql
```

Reports: validity of each operation, operation-type match, selection-structure
precision/recall/F1 (field paths), a strict F1 (field paths + canonical
arguments), and the specific missing/extra paths and argument mismatches.

## The monday.com experiment

`testdata/monday/` contains five complex, real use cases taken from monday.com's
API docs. Each case has:

- `prompt.txt`    — a natural-language request (no schema field names leaked)
- `expected.graphql` — the canonical query from monday's docs (the ground truth)
- `actual.graphql`   — the query a sub-agent produced using **only** the file tree

Each `actual.graphql` was generated by a separate sub-agent that was given only
the prompt and the path to the generated file tree, and told to explore with
shell tools (no internet, no reading the full SDL dump). See `RESULTS.md` for
the outcome.

Reproduce the comparison:

```sh
./testdata/monday/run.sh
```
