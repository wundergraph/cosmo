# Defer battle-test — campaign report

Campaign: 2026-06-20, branch `dazed-geology`.
Duration: ~47 min (budget was 6h; converged early).
Stop reason: **converged** — 3 consecutive rounds with zero new root-cause findings AND the escalation ladder exhausted (the requested "no more bugs for 3 consecutive rounds").
Rounds: 6. Candidate ops generated by the codex fleet and judged by the oracle: **1,023** (plus the seed corpus).

## Method (recap)

- **Codex fleet** (4 stateless workers/round, rotating strategies) generated structure-aware adversarial `@defer` operations, steered each round by an avoid-digest of known findings so it hunted new ground.
- **`oracle.mjs`** judged every op deterministically against 6 spec-free oracles: reconstruction-equivalence (`defer == normal`), terminal-frame, pending-closure, path-validity, error-parity, and crash/hang.
- **Opus (orchestrator/verifier)** independently reproduced every candidate on the wire, deduped, and recorded — nothing was recorded without reproduction.
- The router ran under the **Go race detector** (`-race`) the whole campaign as a concurrency sanitizer.
- **Root-cause dedup**: new query signatures that traced to an already-known defect were not counted as new bugs (signature dedup over-counts; convergence is judged by root cause).

## New bugs found (beyond the pre-existing B1–B7 in `docs/BUGHUNT.md`)

All three are **high** severity and **new beyond B1–B7**. Regression tests in `../tests/defer.bugs.test.ts` (assert correct behavior; fail today, pass once fixed).

### BT-1 — `@defer` on a `__typename`-only fragment combined with a nested in-list defer nulls the parent + drops the terminal frame
- Round 1. Repro: `{ article(id:"a1"){ id title ... @defer { __typename } reviews{ id ... @defer { __typename author{ __typename id displayName } } } } }`
- Normal: 200, full article, 0 errors. Defer: a single frame `{"data":{"article":null}}` with **no `hasNext:false` terminal** → wrong data + client hang.
- A bare root `... @defer { __typename }` is clean; the defect needs the nested in-list combination.

### BT-2 — data race in concurrent `@defer` resolution (engine loader + astjson)
- Round 3, oracle = Go `-race`. **5 race instances, one root cause** (full traces in `race-traces.txt`).
- The defer dispatch (`router/core/graphql_handler.go:354` → `resolve.go:639`) spawns an errgroup of parallel fetches; two goroutines concurrently WRITE (merge a fetch result, `astjson/mergevalues` ← `resolve/loader.go:657/683`) and READ (render fetch input, `astjson/parser` ← `resolvable.go:906` ← `inputtemplate.go` ← `loader.go:1412/1555`) the **same astjson value with no synchronization**.
- Root cause in the engine dependency (`graphql-go-tools/v2 .../pkg/engine/resolve/loader.go` + `astjson`), not this repo. Invisible to sequential happy-path tests.

### BT-3 — nested + overlapping-field defer: announced pending payloads are never delivered, stream closes early
- Round 3. Repro: `{ user(id:"u1"){ id recommendedArticles{ id title } ... @defer { recommendedArticles{ id title ... @defer { heroImageUrl stats{ views } reviews{ id rating } } } } } }`
- Normal: 200, 0 errors, full data. Defer: emits only the initial frame (`hasNext:true`) then closes the multipart stream with **no incremental/completed frames and no terminal** → deferred content never delivered, client hangs.
- Trigger: nested defer where a field is selected both outside and inside the defer.

## Systemic root cause (the headline)

**The router does not correctly deliver multiple / nested / type-conditional / overlapping deferred fragments.** This single underlying defect manifests as several distinct symptoms depending on query shape:
- nulled parent + missing terminal (BT-1),
- deferred payloads never delivered + premature stream close (BT-3),
- dangling `pending` ids + premature `hasNext:false` on abstract-type sibling defers — already filed as B4, and re-confirmed this campaign by round-4 signature `178b1b1c` (`search` union with double-nested type-conditional defers: announces 4 pending, completes 2).

Plus the orthogonal **concurrency** defect (BT-2): parallel deferred fetches race on the shared response document.

Net: single-level `@defer` works well (the 33-case happy-path suite passes), but **nested/sibling/abstract/overlapping defer and concurrent defer are broken in multiple ways.**

## Oracle hardening done mid-campaign
- Gated the protocol invariants (terminal-frame / pending-closure / path-validity) on a valid normal-mode baseline — invalid (syntax-error) queries were false-positiving them.
- Recorded that reconstruction-equivalence is **unsound across an error null-propagation boundary** (defer cannot retroactively null already-sent parent data); the B5-family error sigs were deduped as known rather than counted.

## Limits & honest caveats
- The defer resolver lives in the `graphql-go-tools` engine dependency (outside this repo); BT-2's root cause and most of the nested-delivery logic are engine-side. No router/engine source was patched — findings are characterizations + failing regression tests + this report.
- Convergence means "no new *root causes* in 3 escalated rounds at this corpus/strategy budget," not a proof of absence. A longer run, a real reference-implementation differential oracle (graphql-js incremental delivery), and true concurrency-burst scheduling would extend coverage further.
- Mock resolvers are deterministic, so reconstruction-equivalence is sound for non-error cases.
